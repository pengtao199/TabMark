# AGENTS.md

## 项目开发总规则（强约束）

本项目后续开发必须遵守以下规则，优先级高于临时方案。

1. 必须优先模块化开发
- 所有功能改动优先在 `src/newtab/modules/` 等小模块中完成。
- 禁止把逻辑重新合并回超大单文件。
- 新功能应按职责拆分（bookmarks/search/settings/ui 等），保持可独立迭代。

2. 禁止走“回退大文件”捷径
- 禁止为了临时可用而把入口切回历史大文件/备份文件。
- 禁止通过引入 `*.legacy.js`、`*.bak_*` 作为运行入口解决问题。
- 修复必须发生在当前模块化代码路径中。

3. 现有架构边界（必须遵守）
- `src/newtab/*.js`：页面入口与桥接层（轻量）。
- `src/newtab/modules/*.js`：功能实现层（主开发区）。
- `src/shared/*.js`：跨页面共享能力（存储键、打开方式等）。
- `src/background/*.js`：后台路由与动作处理。
- 禁止在错误层写逻辑（例如把 background 逻辑塞进 newtab 模块）。

4. 变更原则
- 小步提交：每次只做一个明确目标（修错/功能点）。
- 最小影响：优先局部修复，不做无关重构。
- 可追溯：每次改动要能解释“为什么改、影响范围是什么”。

5. 质量门禁（提交前必须执行）
- 语法检查：`node --check`（至少覆盖改动文件）。
- 导入检查：`npm run audit:imports`。
- 若改动启动链路：需额外检查入口文件和被入口直接加载的模块。

6. 启动链路稳定性规则
- 任何会在启动时执行的模块不得出现：
  - 顶层非法 `return`
  - 重复声明（如重复 `function` / `import` 同名）
  - 未定义依赖直接调用
- 对跨模块共享能力，优先通过统一状态桥接（如 `script-runtime-bridge.js`）访问。

7. 文档与规范同步
- 架构调整后要同步更新：
  - `docs/ARCHITECTURE_RULES.md`
  - `docs/FOLDER_STRUCTURE.md`
  - 本文件 `AGENTS.md`
- 若规范冲突，以本文件为准，并在后续文档中消除冲突。

8. Git 流程
- 分支可继续使用当前策略；提交信息需清楚说明范围。
- 禁止提交仅用于临时绕过问题的入口切换。
- 推送前确认工作区无意外临时文件（尤其 `*.legacy.js`、调试脚本）。

9. 协作约定
- 当用户明确要求“按模块化修复”时，必须严格执行，不得改走其它路线。
- 如果修复路径会带来架构级后果，先说明风险再实施。
