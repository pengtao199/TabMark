# 书签面板整理功能计划文档

## 摘要
在现有模块化 newtab 架构下，为书签主面板增加两类能力：

1. 拖拽时可直接把书签或文件夹放入目标书签文件夹。
2. 进入任意文件夹后，在当前列表末尾显示一个虚拟的“+”虚线卡片，点击后可直接新增书签或新增文件夹。

实施顺序按“先稳定数据动作，再补 UI，再补创建入口”推进，确保每一步都不破坏现有启动链路、同级排序、面包屑导航和目录切换。

## 关键实现方案

### 第一阶段：补齐主面板“拖拽放入文件夹”
目标是在不破坏现有同级排序的前提下，支持拖拽放入文件夹。

实现约束：
- 继续沿用现有模块化入口，不回退到任何 `*.bak_*` 或历史大文件。
- 主逻辑放在 `src/newtab/modules/script-sortable.js`。
- 当前目录状态继续使用 `#bookmarks-list.dataset.parentId`。
- 刷新仍走现有 `updateBookmarksDisplay(parentId)`。

实现细节：
- 保留现有同级拖拽排序逻辑。
- 新增“释放目标是否为 `.bookmark-folder`”判定。
- 若拖拽目标是文件夹卡片，则调用 `chrome.bookmarks.move(itemId, { parentId: targetFolderId, index: targetIndex })`。
- `targetIndex` 第一版统一采用目标文件夹末尾，避免内部顺序计算复杂化。
- 若拖拽目标不是文件夹，则沿用原有排序逻辑。
- 增加非法移动保护：
  - 禁止文件夹拖入自己。
  - 禁止文件夹拖入自己的后代目录。
- 移动成功后失效相关缓存并刷新当前目录。
- 移动失败后回退 UI，以 Chrome 真实书签树为准重新渲染。

第一版不做：
- 不做拖拽悬停自动进入文件夹。
- 不做拖进文件夹后内部精确插位。
- 不做文件夹展开态拖拽。

### 第二阶段：增加拖拽放入文件夹的视觉反馈
目标是让用户明确感知“当前是在放入文件夹，不是在普通排序”。

实现位置：
- 交互逻辑放在 `src/newtab/modules/script-sortable.js`
- 样式放在 `src/styles.css`

实现细节：
- 当拖拽项进入文件夹卡片命中区域时，为目标文件夹增加专用状态类，例如 `is-drop-target`。
- 目标文件夹显示更明显的高亮边框、轻微背景变化或轻量缩放。
- 非法目标显示禁用态，避免误解。
- 拖拽结束、取消、离开目标时统一清理状态类。

样式原则：
- 不影响原有卡片尺寸和换行布局。
- 不做过强动画，避免在 `flex` 布局下跳动。

### 第三阶段：在当前目录末尾增加虚拟“+”卡片
目标是在书签主面板提供一个稳定的新建入口。

实现位置：
- 渲染逻辑放在 `src/newtab/modules/script-bookmark-list.js`
- 样式放在 `src/styles.css`

实现细节：
- 在当前目录真实书签渲染完成后，额外 append 一个虚拟卡片。
- 虚拟卡片不属于 Chrome 书签数据，不写入缓存，不参与排序。
- 样式为虚线边框、加号图标、明显的可点击态。
- 卡片固定显示在当前目录最后一个位置。
- 所有目录统一显示，包括书签栏根目录和子目录。

必要限制：
- 该卡片不得参与 `Sortable` 的真实拖拽排序。
- 不得进入 `bookmarkOrderCache` 或任何真实节点集合。
- 目录刷新时只能追加一次，避免重复渲染。

### 第四阶段：点击“+”后可新增书签或新增文件夹
目标是在当前目录中直接完成创建动作。

建议新增模块：
- `src/newtab/modules/script-bookmark-create.js`

配套位置：
- 创建弹窗结构放到 `src/index.html`
- 样式放到 `src/styles.css`
- 通过 runtime bridge 暴露必要方法，维持现有模块协作方式

实现细节：
- 点击虚拟“+”卡片后，打开一个轻量创建面板或弹窗。
- 弹窗包含两个模式：
  - 新增书签
  - 新增文件夹
- 当前目录 `parentId` 从 `#bookmarks-list.dataset.parentId` 获取。
- 新增书签调用：
  - `chrome.bookmarks.create({ parentId, title, url })`
- 新增文件夹调用：
  - `chrome.bookmarks.create({ parentId, title })`
- 创建成功后：
  - 关闭弹窗
  - 失效当前目录缓存
  - 刷新当前目录列表
  - 保持面包屑和当前目录不跳转
- 创建失败后：
  - 显示错误反馈
  - 不修改现有列表状态

表单要求：
- 新增书签：标题、URL
- 新增文件夹：标题
- URL 做最基本合法性校验
- 防止重复提交

### 第五阶段：缓存同步、边界兜底与文档同步
目标是把功能收口到长期稳定可维护状态。

缓存与状态处理：
- 拖入文件夹后，源目录和目标目录缓存都应失效。
- 新增书签或文件夹后，当前目录缓存失效。
- 渲染时保证虚拟“+”卡片始终在真实数据之后单独附加。
- 若 `Sortable` 因虚拟卡片导致索引异常，需要显式过滤该卡片。

需要补的边界：
- 目标文件夹不存在时的拖拽失败处理。
- 连续快速拖拽的并发保护。
- 创建弹窗打开时再次点击的防重处理。
- 当前目录为空时，仍正常显示虚拟“+”卡片。
- 当前目录只有文件夹或只有书签时，布局正常。

文档同步要求：
- 若新增 `script-bookmark-create.js`，更新 `docs/FOLDER_STRUCTURE.md`。
- 若补充了新的面板交互规则，更新 `docs/ARCHITECTURE_RULES.md`。
- 若后续实际实现中引入新的协作约束，再同步 `AGENTS.md`。

## 公共接口与行为变更
这次不引入 background 新接口，主要使用现有 Chrome Bookmarks API。

会新增或调整的前端行为接口：
- 主面板拖拽行为从“仅同级排序”扩展为“同级排序 + 放入文件夹”。
- 书签列表渲染行为从“仅真实节点”扩展为“真实节点 + 一个虚拟创建卡片”。
- 新增一个前端创建模块，对外暴露：
  - 打开创建弹窗
  - 根据当前模式创建书签或文件夹
  - 创建成功后刷新当前目录

默认实现选择：
- 拖入文件夹时插入目标文件夹末尾。
- “+”卡片点击后先展示二选一创建入口，而不是分别放两个固定卡片。
- 新建完成后停留在当前目录，不自动进入新建文件夹。

## 测试与验收

### 静态与启动链路检查
- 对改动文件逐个执行 `node --check`
- 执行 `npm run audit:imports`
- 若改到入口直接加载链路，额外检查入口模块导入关系是否正常

### 功能验收用例
1. 在当前目录拖动书签到另一个文件夹卡片上，书签成功移入目标文件夹。
2. 在当前目录拖动文件夹到另一个文件夹卡片上，文件夹成功移入目标文件夹。
3. 将文件夹拖到自己身上，被阻止。
4. 将文件夹拖到自己的子孙文件夹，被阻止。
5. 普通拖动到非文件夹区域时，仍执行原有同级排序。
6. 拖过文件夹时，有明确高亮反馈。
7. 拖拽结束后，高亮态正确消失。
8. 当前目录末尾始终显示一个虚拟“+”卡片。
9. “+”卡片不参与真实排序，不进入真实缓存。
10. 点击“+”后可选择新增书签或新增文件夹。
11. 在当前目录新增书签后，书签立即显示在当前列表中。
12. 在当前目录新增文件夹后，文件夹立即显示在当前列表中。
13. 新建失败时不污染当前列表。
14. 切换目录、面包屑导航、默认文件夹切换功能保持正常。
15. 当前目录为空时，仍能看到“+”卡片并成功创建内容。

## 假设与默认决策
- 默认只支持“拖拽松手后放入文件夹”，不做悬停自动进入文件夹。
- 默认放入目标文件夹末尾，而不是插入到目标文件夹的特定顺序位置。
- 默认“+”卡片在所有目录都显示。
- 默认点击“+”后使用一个统一弹窗承载“新增书签 / 新增文件夹”。
- 默认新建完成后留在当前目录，不自动跳转到新建文件夹内部。
- 默认不改 background 路由，不新增存储结构，只复用现有 `chrome.bookmarks` 能力。
